# 客户正文正向写作模式

本文件只服务 Task 3 正文生成。它提供可组合的写作动作，不提供项目事实、固定句式或章节模板；所有具体内容仍来自当前 compiled brief。

## 一章的说服任务

先用一句客户判断说明本章要解决什么、为何影响选择，再依次交付：

1. 项目依据：来自 Requirement、客户需求或已授权 Evidence 的具体事实；
2. 方案选择：本章采取的关键取舍及其与核心命题的关系；
3. 实现机制：动作、责任、时点、资源边界、协作接口和回退路径；
4. 可见成果：评委此刻就能检查的表格、清单、样例、分镜、节奏表或看板；
5. 判断交接：本章证明了什么，以及下一章将在此基础上继续证明什么。

每章形成一个独有认知增量。核心记忆句负责统一方向，章节内容负责逐步增加理由与证据。

## 段落构造

优先采用以下正向构造，按内容需要组合：

- **判断 → 依据 → 含义**：先给项目判断，再给可核验依据，最后说明它怎样改变方案选择。
- **客户摩擦 → 方案动作 → 可感知变化**：从具体角色或使用环节进入，让动作对应一个明确变化。
- **动作 → 责任 → 时点 → 检查**：把方案写成可执行、可追踪的交付机制。
- **目标 → 口径 → 数据源 → 复盘动作**：让指标直接指导下一轮决策。
- **风险触发 → 安全回退 → 升级路径**：让妥帖性成为方案的一部分。
- **拟议样例 → 字段实值 → 适用边界**：让 illustrative 成果既具体又诚实。

一个段落优先完成一个评委判断。主张之后紧跟它成立所需的依据、机制或成果，段落结尾落到客户结果或检查方式。

## 客户可见成果

把 `visible_outputs` 直接写成正常方案内容：

- 矩阵用于展示多个对象之间的精确对应；
- 清单用于展示责任、时点、输入和完成标准；
- 样例用于让评委提前体验拟议交付；
- 节奏表用于展示阶段推进与检查点；
- 看板用于展示指标口径、数据源和复盘动作。

逐项填入 `required_fields` 的项目特定实值，并让标题、字段和说明共同表达 `purpose`。版式服从信息关系和客户阅读效率。

全案只把 `display_role=signature` 的成果作为主亮点：先交代它替评委解决的判断，再让成果本身占据最清晰的版面。`supporting` 提供必要证明，`reference` 服务快速查验，使“完整”与“重点突出”同时成立。

## 语言质感

- 以具体对象和动作动词承载句子，例如“核对入口有效期”“汇总高频问题”“在里程碑签收”。
- 用客户结果开启或收束关键段落，让我方能力通过机制、责任和成果自然显现。
- 专业术语紧邻其对象、作用和检查方式，使术语承担信息而非气氛。
- 长短句交替；标题表达判断，正文完成推导；重复信息用表格或交叉引用压缩。
- 政府与公共项目突出正当性、连续性、可审计性和协同负担；商业项目突出行为摩擦、生意机制、验证口径和资产沉淀。

## 真实性表达

事实采用 brief 中的安全投影；拟议动作使用与 `intended/planned` 一致的方案语气；待确认资源写清适用条件和 owner；illustrative 内容明确为本项目拟议样例。每个具体数字、案例、能力和承诺都能回到当前 brief 的 source anchor。

交付前，从客户视角复述三件事：本章的新判断、支持判断的最强依据、客户能当场检查的成果。三者清楚且彼此相连，即可交付独立审计。
