# PPT 结构与视觉叙事模式

本文件服务 presentation planner。结构稿把定稿正文压缩成逐页判断与画面任务；下游图片工作流再确认风格、图像后端和一页样张。

## 演示模式：视觉分镜式（默认）与演讲式
先确认这套 PPT 有没有人现场讲：
- **视觉分镜式（默认，无人演讲 / 图片 PPT）**：甲方自己翻阅、下游图片工作流逐页出图。减轻阅读负担靠的**不是页少，而是页多、每页信息量小、每页一个明确的视觉画面**。能拆的都拆开——一个概念总述之后，它的每个子部件（如 BSB 的每道工序、每个案例、每类设备、每名核心角色）各用一页画面单独呈现。每页 render_text 短：结论标题 + 少量关键指标/标签，其余交给画面。**宁多勿少**（页多了用户可删，缺页更难补）。
- **演讲式（仅当明确有人讲标时）**：每页一个判断、上屏更精简，靠讲解补全。
默认视觉分镜式。两种都只有一张 signature 页、都遵守证据图红线与 core/appendix 分轨。

## 一页的最小单位
每张核心页同时回答五件事：
1. 读者（甲方）此刻要形成什么判断；
2. 哪一句结论作为标题；
3. 哪些准确文案必须上屏；
4. 什么画面最适合完成这个判断；
5. 它从上一页承接什么、向下一页交出什么。

标题表达结论，画面承担主要证明。上屏文案按模式：视觉分镜式每页信息量小、只留结论标题 + 少量关键指标/标签，靠画面承载；演讲式只留最短解释。表格适合比较、核验与附录；情绪、洞察和主张优先用主视觉、场景或关系图表达。

## 三种常用叙事骨架
**战役与传播提案**：客户挑战 → 项目洞察 → 核心命题 → signature KV/成果 → 传播机制 → 阶段节奏 → 渠道与动作 → 数据/案例证明 → 交付保障 → 收束。把记忆句和标志性画面置于前半段，后续动作持续解释它如何发生。

**品牌、空间与招商叙事**：场所原点 → 独特张力 → 定位主张 → 人群与生活方式 → 空间/业态机制 → 代表性场景 → 合作价值 → 资源与联系。用大图、留白、短标题与图像序列，让照片与场景承担信任。

**活动、美陈与体验方案**：项目背景 → 主题推导 → 核心主题 → 主视觉 → 关键体验/装置 → 传播预热 → 现场路径 → 资源 → 施工与应急 → 附录效果图/尺寸/媒体清单。核心提案保留最能解释主题、体验和可执行性的页面；同类效果图、细节图、尺寸图和长名单形成有序的 visual proof series 进入附录。

这些骨架是候选节奏，实际页序服从本标评分项、主线、signature 与真实素材。

## 核心轨与附录轨
- `core`：完成从客户张力到选择理由、主亮点、执行与可信度的最短说服链，含 1–2 张决定性效果图。
- `appendix`：承接 mandatory 对照、详细排期、媒体/资源长名单、尺寸材质、**多张效果图系列**、预算口径、风险预案和查验材料。

核心页数由必要判断决定。相邻页产生新的认知增量；重复视觉只用于有意设计的系列展示，并保持同一版式节奏。**大量效果图住在附录轨，不稀释 core 的说服链。**

## 页型与画面任务
| 页型 | 主要画面 |
|:---|:---|
| context / tension | 场景照片、对象关系、前后对比、关键事实组合 |
| insight / thesis | 单一隐喻、因果图、强结论与一处视觉锚 |
| signature | 最大层级的核心成果、KV、体验场景或项目化样例 |
| mechanism / process | 路径图、泳道、闭环、系统关系 |
| roadmap / timeline | 阶段节奏、关键事件、波次与交付节点 |
| data_evidence / case_study | 数据图、证据截图、案例前后关系 |
| visual_proof / asset_gallery | 效果图、分镜、样稿、空间视角或有意的系列展示 |
| resources / team / risk | 责任图、资源组合、检查点、应急卡 |
| compliance / appendix | 对照表、参数、清单、尺寸、预算和可快速查验材料 |

## Signature 页
全案只有一个 signature 页，承担全案制胜一击，落在**制胜轴**上：**管理制胜型**用最清楚的版面展示核心成果 / 机制图；**创意制胜型**就是核心创意 KV / 大 idea 的主视觉——这一版面本身就是那个想法，而不是再证一次方法或流程。把它同时作为图片工作流的一页样张，让后续页面继承同一视觉身份。

## 视觉系统与素材请求
结构稿先定义稳定的视觉概念、配色方向、字体层级、图像语言和版式节奏；每页再选符合语义的构图。素材请求分三类：
- `generate`：由图片模型生成场景、插画、抽象视觉或信息图底图；
- `strict_input`：必须保留真实内容的 Logo、效果图、数据图、截图、案例图或技术图；
- `style_reference`：只提供色彩、排版、质感和图像语言参考。

`available` 素材给真实可访问路径；`needs_user` 写清所需内容与角色；`generate` 给具体正向画面描述与规避元素。`rights_status` 让下游生成前确认使用边界。

## 证据图红线
`generate` 只允许效果图、示意图、信息图这类**对未来方案的想象**。**过往案例现场照、资质证书、荣誉、数据截图等证据类画面永不 generate**——它们只能 `strict_input` + `needs_user` 走真实素材（标 `evidence=true`），缺失时进 `_风险与待核实.md`。理由：虚构的文字可以核实替换，而生成的假证据图一旦上屏就是直接造假，性质完全不同。

## 完整商业稿：缺失素材用意向图顶位
结构稿永远输出**读起来完整的商业 PPT**——任何一页都不能因为真实素材还没到位而露出缺口，更不能在页面上向读者要资料。
- **上屏零索要**："待提供/待补充/请上传/素材缺失/待替换/占位"一律不上屏（标题、要点、标签、takeaway 都算上屏）。
- **画面不留空位**：`prompt_seed`/`main` 里不写"真实图位置留白/预留/待贴"——**顶位视觉把版面做满**：用 `generate` 的意向图、图标、风格化示意完成该页的商业表达（设计语言明显、`avoid` 注明不得仿冒真实照片/证件/公章/系统截图），真图到位后**整图替换**，而不是在图里抠个洞等着贴。
- **替换槽保留**：原证据位仍以 `strict_input + status=needs_user + evidence=true` 留在该页，顶位素材用 `stand_in_for` 指向它。
- **缺失只走内部通道**：页级 `unverified_notes`、outline 末尾的"素材替换清单"、`_风险与待核实.md`。读者看到完整方案，你按清单换真图。
- 与证据图红线不冲突：顶位是明显的意向/图标，不冒充证据；真证据仍只能真实素材。

## 下游消费契约
blueprint 顶层 `deck.field_contract` 声明两类字段：`on_screen`（`title`/`render_text`/`visual` 渲染层——页面上只允许出现这些）与 `internal_only`（`unverified_notes`/`truth_boundary`/`source_refs`——**绝不渲染上屏，也绝不写进讲稿/speech**；它们只服务替换清单与内部核对）。`audience_takeaway` 与 deck 级文案虽不直接上屏，但会进讲稿、随每页 prompt 下发，**同属客户面**——一样干净。下游图片/成稿工作流必须按此契约消费，否则内部披露语会漏进客户稿。

## 交给图片工作流
把 `outline.md` 作为页序确认材料，把 `deck-blueprint.json` 的 deck context、每页 local context、上屏文案、构图和素材请求映射为逐页图像任务。图片工作流继续完成视觉风格确认、图像后端确认、signature 样张批准、逐页生成、视觉 QA 和 PPTX 装配。主视觉与效果图延展可把 `kv-brief.md` 交给 kv-studio，以 signature KV 为参照图保证整套一致。
