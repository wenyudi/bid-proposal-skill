# PPT 结构与视觉叙事模式

本文件只服务 presentation planner。结构稿负责把已审计方案压缩成逐页判断与画面任务；下游图片工作流再确认风格、图像后端和一页样张。

## 一页的最小单位

每张核心页同时回答五件事：

1. 评委此刻要形成什么判断；
2. 哪一句结论作为标题；
3. 哪些准确文案必须上屏；
4. 什么画面最适合完成这个判断；
5. 它从上一页承接什么，并向下一页交出什么。

标题表达结论，画面承担主要证明，正文只保留观看画面所需的最短解释。表格适合比较、核验与附录；情绪、洞察和主张优先用主视觉、场景或关系图表达。

## 三种常用叙事骨架

### 战役与传播提案

客户挑战 → 项目洞察 → 核心命题 → signature KV/成果 → 传播机制 → 阶段节奏 → 渠道与动作 → 数据/案例证明 → 交付保障 → 收束。

适合把一个记忆句和一个标志性画面置于前半段，再让后续动作持续解释它如何发生。

### 品牌、空间与招商叙事

场所原点 → 独特张力 → 定位主张 → 人群与生活方式 → 空间/业态机制 → 代表性场景 → 合作价值 → 资源与联系。

适合使用大图、留白、短标题与图像序列，让照片、空间和场景承担信任。

### 活动、美陈与体验方案

项目背景 → 主题推导 → 核心主题 → 主视觉 → 关键体验/装置 → 传播预热 → 现场路径 → 资源 → 施工与应急 → 附录效果图/尺寸/媒体清单。

核心提案保留最能解释主题、体验和可执行性的页面；同类效果图、细节图、尺寸图和长名单形成有序的 visual proof series，进入附录或查验轨。

这些骨架是候选节奏。实际页序服从本标的 DecisionJob、评分项、signature 和真实素材。

## 核心轨与附录轨

- `core`：完成从客户张力到选择理由、主亮点、执行与可信度的最短说服链。
- `appendix`：承接 mandatory 对照、详细排期、媒体/资源长名单、尺寸材质、多张效果图、预算口径、风险预案和查验材料。

核心页数由必要判断决定。相邻页面应产生新的认知增量；重复视觉用于有意设计的系列展示，并保持同一版式节奏。

## 页型与画面任务

| 页型 | 主要画面 |
|:---|:---|
| context / tension | 场景照片、对象关系、前后对比、关键事实组合 |
| insight / thesis | 单一隐喻、因果图、强结论与一处视觉锚 |
| signature | 最大层级的核心成果、KV、体验场景或项目化样例 |
| mechanism / process | 路径图、泳道、闭环、系统关系 |
| roadmap / timeline | 阶段节奏、关键事件、波次与交付节点 |
| data_evidence / case_study | 数据图、证据截图、案例前后关系 |
| visual_proof / asset_gallery | 效果图、分镜、样稿、空间视角或有意的系列展示 |
| resources / team / risk | 责任图、资源组合、检查点、应急卡 |
| compliance / appendix | 对照表、参数、清单、尺寸、预算和可快速查验材料 |

## Signature 页

全案只有一个 signature 页。它直接映射 canonical `signature_output_ref`，用最清楚的版面展示已经填实的成果字段，并承担核心命题的主要视觉证明。建议把它同时作为图片工作流的一页样张，以便后续页面继承同一视觉身份。

## 视觉系统与素材请求

结构稿先定义稳定的视觉概念、配色方向、字体层级、图像语言和版式节奏；每页再选择符合语义的构图。

素材请求使用三类：

- `generate`：由图片模型生成场景、插画、抽象视觉或信息图底图；
- `strict_input`：必须保留真实内容的 Logo、效果图、数据图、截图、案例图或技术图；
- `style_reference`：只提供色彩、排版、质感和图像语言参考。

`available` 素材提供真实可访问路径；`needs_user` 写清所需内容与角色；`generate` 给出具体正向画面描述。`rights_status` 让下游在生成前确认使用边界。

## 交给图片工作流

把 `outline.md` 作为页序确认材料，把 `deck-blueprint.json` 中的 deck context、每页 local context、上屏文案、构图和素材请求映射为逐页图像任务。图片工作流继续完成视觉风格确认、图像后端确认、signature 样张批准、逐页生成、视觉 QA 和 PPTX 装配。
