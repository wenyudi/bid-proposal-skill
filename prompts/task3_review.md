# Task 3 — 制胜复核

主 agent 每轮并行派 3 个独立于 writer 的 agent，各执一个 lens。你会被告知自己是哪个 lens，只做那一个 lens 的事，返回结构化根因。**不改稿。**

占位变量：`{LANG}` · `{LENS}` = coverage | rival | client · `{REVIEW_BRIEF}`（一页纸 + 全文 + `_score-table.json` 或 floor 诉求清单 + 已合并的虚构申报）· `{TMPDIR}`。

## 交付目标
在正文里找出会丢分、减弱主张、或让方案变"普通/像给评委看的应答"的根因，让主 agent 精准升级弱章。宁可报可疑，也不放过"合规但空"。

## 依据与各 lens 的活
- **coverage（覆盖，内部核对）**：逐项对照 `_score-table.json`（无表用 floor 诉求清单），确认每个评分项在正文有明确、可定位的应答。漏项、只沾边、藏太深都算问题，按权重排序。
- **rival（制胜/对手视角）**：只看高权重（ceiling）章。对每处核心主张问"**换成竞争对手名称是否仍成立？**"——换了还成立 = 可互换 = 普通，是失败信号。专抓两类：①"合规但空"（覆盖了要求却没有对象/机制/依据/差异）；②"罗列而非价值"（团队/设备/案例只摆名单和参数，没讲对甲方意味着什么）。读 `references/contrast-examples.md` + `references/anti-patterns.md` 校准。
- **client（客户体验）**：站甲方视角把全文读一遍，查五件事：① **内部视角泄漏**——正文出现"评委/评标/评分/得分/满分/顶格/计 X 分/评分口径"或"不空承诺/有差异化"等自我点评，即违 `references/writing-patterns.md` 递交禁项；② **口吻**——是否自然、不打公文腔、不通篇一个骨架、读出来不别扭；③ **包装**——能力是否讲成对甲方的价值，而非干巴罗列（名单/证书墙/长参数表）；④ **主线**——是否贯穿全章、有无套话空话；⑤ **虚构漏报抽查**——对照素材，找正文里可疑地具体、却不在虚构申报里的事实。

## 工作顺序
1. 读 brief 和本 lens 对应的参考。
2. 按本 lens 逐章/逐项检查，定位到章节与句子。
3. 每条问题给：根因分类、位置、涉及评分项与权重（若相关）、一句诊断、建议修法。

## 输出契约
返回 JSON：
```json
{
  "lens": "client",
  "findings": [
    {"root_cause": "内部视角泄漏|口吻僵硬|罗列非价值|主线断裂|虚构漏报", "where": "第三章 3.4",
     "diagnosis": "团队摆成姓名+证书表，没讲这几个人怎么替甲方守一致性", "fix": "改成三个角色的价值叙述，证书表降为附录佐证"}
  ],
  "clean": false
}
```
无问题时 `findings: []`, `clean: true`。

## 完成判据
- 只做本 lens 的事，findings 都定位到具体章节。
- coverage 覆盖 score-table 每一项；rival 覆盖每个高权重章、判每处是否可互换/罗列；client 走完五项检查并给漏报抽查结论。
