# 完成第一份方案

本教程带你用默认 v3 流程完成一份标准技术标草案。结束时，你会知道怎样启动 proposal、怎样确认真实边界和一页纸策略、怎样处理定稿红队，以及怎样判断输出能否进入最终排版。

## 开始前

你需要：

- 已按[安装与注册](../how-to/install-and-register.md)启用 proposal；
- 一份 PDF、DOCX、Markdown、TXT 标书，或可以直接粘贴的标书全文；
- 可选的公司资质、团队、报价和真实案例材料；
- 一个能让宿主联网检索并抓取原文的会话。

本教程假设标书位于 `/绝对路径/示例项目/招标文件.pdf`，素材位于 `/绝对路径/示例项目/投标素材/`。请替换为你的真实路径。

## 1. 启动标准 v3 流程

新开会话并输入：

```text
/proposal /绝对路径/示例项目/招标文件.pdf /绝对路径/示例项目/投标素材/
```

不要添加 `-auto`。第一次使用时保留人工关卡，便于看清哪些判断来自标书，哪些必须由你确认。

proposal 会先检查是否有标书，由工具生成五份确定性空骨架，再让 Task 1 拆解 mandatory、评分项、预算和交付物，同时建立客户角色、需求、判断标准、候选价值和交付边界。机械 schema 不再占用策略师的主要注意力，此时还不会直接写完整正文。

## 2. 完成 Gate 1

Task 1 后，你会先看到一次策略总览，然后每轮只收到一个问题，结构类似：

```text
决策：核心团队资源边界
问题：项目经理和关键岗位是否已确认可在合同期投入？
为什么现在要定：未经确认就写成确定配置，会形成真实性和履约风险。
AI 推荐：先按已提供证明材料确认，其余岗位保持 intended。
```

按事实回答，例如：

```text
项目经理已确认；其他岗位只有候选名单，先不要写成已锁定。
```

回答会同时更新实际受影响的价值主张、Claim、Action、资源和验收边界，而不只是把问题标为“已确认”。后续问题仍一次一个。你可以补充条件，但不要为了推进流程确认尚不存在的能力或授权。

当决策前沿、被阻塞决策和迷雾都清零后，proposal 会给出能力与承诺边界摘要并进入研究；此时还不会要求你批准最终创意策略。

## 3. 观察研究后的亮点选择

Task 2 会围绕本标的实际证据缺口抓取公开原文，并把 Evidence 原记录与“它能证明什么”分开。若新证据改变了命题或价值机制，它会显式重新打开候选空间。Task 2.5 才从更新后的候选池选择 lead、supporting 和 reserve：

- lead 是本案最值得被记住、且证据与交付链都足够完整的客户价值；
- supporting 补足 lead 无法单独覆盖的重要判断；
- reserve 留在内部，不为了显得丰富而塞进客户稿。

Task 2.5 还会为每个 lead 指定一个最小客户可见成果（例如已填样例、分镜、节奏表或看板），并从全部成果中只选一个 signature 主亮点。它只约束评委必须看到的字段与真实性边界，不把正文变成内部检查表或平均用力的成果堆叠。

如果研究发现新增成本、能力或授权边界，流程只会重新打开受影响的 Gate 1 问题。它不会另加一套固定关卡，也不会用文案绕过缺口。

## 4. 确认一页纸策略

Task 2.5 收敛后，你只需要做一次高价值检查。proposal 会展示：客户张力、尖锐洞察、最终命题与最强替代、让最终命题胜出的依据、接受的取舍、改选信号、核心记忆句、洞察到执行/证明的推导、名称互换结果、signature 主亮点、落地可信度、五维弱项和逐章贡献。

只回答“按这一页进入写作”，或指出一个需要重做的根因。例如：

```text
先不要写。洞察仍是行业常识，回到客户参与入口与回访断点重新收敛。
```

确认批准的是方向和取舍，不是每个句子的最终措辞。使用 `-auto` 时这一页只会被记为 assumed，因此生成物仍不可直接递交。

## 5. 等待分章写作与独立审计

达到 safe-draft generation gate 后，所有章节从同一份快照获得各自的最小 brief。每份 brief 都带同一页策略、完整章节 spine、本章贡献和前序骨架摘要；章节 writer 看不到 private 原文和无关内部图谱，只写 Markdown。

每章写完后，另一个未参与写作的 auditor 会检查：

- 对应 Requirement 是否在正文中被实质回答；
- Claim 和 Action 是否被文本准确表达；
- scope、承诺强度和证据是否被扩大；
- 是否出现 brief 白名单之外的新主张。
- required 客户可见成果是否真正填实全部字段，而非只写“将建立”。

只有状态为 `valid` 的正式章节才能进入方案综述。你无需手工把内部 ID 或审计标签写进正文。

## 6. 完成 Gate 2

装配后，profile 会启动 1/2/4 个红队角色。策略批评者会独立做一句话复述、名称互换、主亮点、逐章主线、落地和阅读效率测试，并精确引用正文；只有当前 snapshot 且能在当前报告定位的引用才进入 customer-fit。硬性致命问题先回 owner 修复，其余问题在 Gate 2 一次处理一条。

每题会给出推荐处置及得失：

- `采纳修改`：问题成立，修改表达或上游事实；
- `保留并补证据`：方向有价值，但当前证据不足；
- `不采纳`：有明确依据证明是误报，并保留处理记录。

如果修改了任一 canonical，当前快照绑定的章节、摘要和审计都会重新生成或复验。全部处理完后确认定稿。

## 7. 可选：生成图片 PPT 结构稿

如果最终需要图片版 PPT，可在 Gate 2 后增加一次 presentation 结构压缩。proposal 会输出 `_PPT生产包/outline.md` 与 `deck-blueprint.json`：前者用于确认页序、signature 和素材映射，后者交给 image2 等图片工作流。先批准 signature 样张，再生成其他页面；详细步骤见[生成 PPT 结构稿并交给图片工作流](../how-to/create-ppt-handoff.md)。

结构稿通过表示内容和页序可审阅，图片中的中文、数字、Logo、效果图和技术图仍需逐页视觉 QA。

## 8. 检查交付结果

最终汇报会给出实际卷册路径，并明确 `submission_ready`。打开卷册后应看到：

```text
<方案标题>-<时间戳>/
├── 技术方案-完整版.md
├── 分册/
├── _内部研判.md
├── _人工待办.md
├── _PPT生产包/              可选
├── _acceptance-receipt.json
└── _state/
```

先检查以下三点：

1. receipt 明确 `delivery_status=submission_ready`，state/report hash 对应当前文件；
2. `_人工待办.md` 没有未解决的真实性、mandatory、预算、授权或占位符硬项；
3. `技术方案-完整版.md` 中的投标人名称、日期、资质、人员和报价已经由人工核验。

只有 `技术方案-完整版.md` 和需要的 `分册/` 内容可进入客户版排版。`_内部研判.md`、`_人工待办.md`、`_state/` 和 `_state.last-good` 永远不递交。
