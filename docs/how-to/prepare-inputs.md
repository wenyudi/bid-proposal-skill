# 准备标书与投标素材

输入越清楚，Gate 1 越少重复追问，证据和承诺边界也越安全。开始运行前按本页整理一次即可。

## 1. 准备必需的标书输入

proposal 接受以下任一形式：

- 本地 PDF、DOCX、Markdown 或 TXT 路径；
- 直接粘贴的标书全文。

优先提供包含以下内容的完整版本：采购需求、评分办法、资格与实质性条款、预算、交付清单、格式要求、澄清或更正。不要只提供项目名称或几页需求摘要。

如果标书由多个文件组成，把所有相关路径放在同一次请求中，并说明哪个是主文件、哪个是澄清或附件。

## 2. 将辅助材料按事实类型整理

建议使用一个独立目录：

```text
投标素材/
├── 01-企业资质/
├── 02-类似业绩/
├── 03-团队与人员/
├── 04-报价与资源/
├── 05-品牌与能力资料/
└── 06-沟通与踏勘纪要/
```

文件名尽量写明材料性质、日期和版本。对于合同、证书、验收、截图等材料，补充其具体可证明范围；“有材料”不等于已经核验，也不等于获得对外发布授权。

## 3. 标记 private 纪要

沟通、踏勘、答疑会和售前笔记必须在请求中标为 `[notes]`：

```text
/proposal /路径/招标文件.pdf /路径/企业材料/ [notes] /路径/踏勘纪要.md
```

`[notes]` 只用于内部校准需求、偏好和风险。原句、内部关系、底价、个人意见或未公开信息不得进入客户稿；即使后来找到公开依据，也只能使用独立的公开投影。

## 4. 单独维护可复用案例

长期复用的公司案例放入仓库 `casebase/`，而不是每次混在临时素材中。每个非 `_` 开头的 Markdown 案例会被自动发现。

案例必须写清我方角色、实际范围、核验材料、可见性和允许用途。具体操作见[维护案例库](manage-casebase.md)。第三方案例可以用于 benchmark 或可行性说明，不能证明我方能力。

## 5. 在请求里说明本次目标

除了路径，可附上已经明确的限制，例如：

```text
/proposal /路径/招标文件.pdf /路径/投标素材/ -deep
本次只做技术标；报价上限以标书为准。团队名单尚未最终确认，案例客户名不得公开。
```

这些说明不会替代材料核验，但能让 Task 1 更快把问题归入事实、研究缺口、人工决策或范围外事项。

## 提交前自检

- 标书路径真实存在，且不是只有项目名的空文件；
- 评分办法、mandatory、预算、交付物和澄清文件齐全；
- 资质、案例、团队、报价分别有可定位的材料来源；
- private 内容已标 `[notes]`；
- 未授权客户名、Logo、评价、金额和数字结果没有被默认视为可公开；
- 已明确本次只覆盖技术标，法定模板另行处理。

准备完成后，按[选择运行模式](choose-modes.md)组合调用参数。
