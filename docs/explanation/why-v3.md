# 为什么是 v3

v3 的变化不是“多写几张表”，而是把投标方案从一次性文案生成，改成一条可验证的客户判断链。它要解决的核心矛盾是：方案既要有创意和记忆点，又必须合规、可信、能交付，还要让评委读起来顺畅。

## 直接写稿为什么不够

从标书直接生成章节，容易出现五类错位：

1. **结构覆盖代替实质回答**：评分项被映射到某章，但正文只提到关键词，没有回答评委要判断的内容。
2. **亮点脱离客户**：创意看起来新，却说不清对哪个角色、哪个结果、哪条判断标准有价值。
3. **证据与承诺混写**：行业案例、我方案例、目标和保证被放在同一语气层，读起来有力，核验时却站不住。
4. **底层严谨破坏阅读**：为证明覆盖而把 ID、状态、审计标签和限定语塞进客户稿，导致方案像机器检查表。
5. **合规但空**：Requirement 和流程都写对了，却没有样例、分镜、节奏表或看板等评委可以直接判断的项目成果。

legacy 2.x 保留了装配、合规、QA 和红队能力，适合紧急维护旧项目，但没有 v3 的客户价值 canonical、交付 canonical、compiled context、snapshot 和独立 realization 保证。因此它只能显式回退，不再作为自动兜底。

## v3 的四层分工

v3 把不同性质的问题分开处理：

```text
硬约束层       标书要求、法律、预算、真实性、授权
客户判断层     谁在判断/使用/担责，希望得到什么，以什么标准相信
兑现层         Evidence、Claim、Action、责任、资源、验收
表达层         叙事、标题、场景、节奏、表格和方案综述
```

层级顺序很重要。表达层可以让方案更有吸引力，却不能重写前三层；customer-fit 可以指出短板，却不能抵消硬约束。这样，创意获得了清楚的活动空间，而不是被合规条款和内部审计混在同一段文字里。

## 为什么先发散、后研究、再选择

过早要求每个候选都证据齐全，会让方案只剩“我们以前做过什么”，难以产生新的客户价值。完全不设边界地发散，则会把未知能力、成本和承诺写进正文。

v3 采用分阶段收敛：

1. Task 1 从 outcome、efficiency、risk、visibility、experience、asset、contrarian 等镜头生成候选；
2. 不可挽救的违法、负偏离或确认不可行项才立即淘汰；
3. Task 2 补 Evidence 和反证；
4. Task 2.5 用窄硬门、关键短板、Pareto 和组合充分性选 lead / supporting；
5. reserve 留在内部，不挤占评委注意力。

因此，严格底座不会把发散变成模板填空。它推迟的是承诺和发布，不是想象；收紧的是最终选择，不是候选生成。

## 为什么要两道人工关卡

模型能读标书、查公开资料、比较方案，但不能替投标人知道：团队是否真的可用、某项资源能否免费提供、报价底线是什么、案例能否署名、关键 KPI 能否承诺。

Gate 1 把这些真实边界放在写作前确认，避免成稿后才发现整条创意路径不可执行。Gate 2 在报告级硬检和 profile 自适应红队后处理取舍，避免把所有模型质疑都自动改进正文。

两道 Gate 都一次一题，并给推荐与得失。这样既保留人的授权责任，也避免一次抛出几十个问题造成决策疲劳。

## 为什么客户稿反而更轻

五份 canonical、DecisionPath、成果契约和审计结果只在底层存在。writer 接收的是本章最小 brief，客户稿不显示 Role、Need、DecisionJob、Claim、Evidence/OUT ID、状态机或 fit。

这使最终方案可以只呈现：

- 以客户任务或结果开篇的自然标题；
- 少而强的价值组合；
- 与创意紧邻的执行、责任、时点和验收；
- 必要的集中表格与清楚口径；
- 每个 lead 亮点至少一个自然融入正文、字段已填实的可检查成果；
- 不夸大、不堆限定语的顺滑正文。

严格性被用来保护阅读体验，而不是展示系统有多复杂。

## v3 的代价

v3 更适合重点政企标，但代价真实存在：

- 写作前需要更完整的状态建模和 Evidence；
- canonical 一旦改变，当前实现会使全部 snapshot-bound 章节和摘要失效；
- 独立审计增加 agent 调用和时间；v3.4 用 writer-only Markdown、2–3 章批量 audit、复用 strategy critic 生成 report-anchored fit judgments，并把 PPT 生产留在可选派生阶段控制增量；
- 资料不足时，系统更可能给出不可递交草案，而不是“看起来完整”的成稿。

这些代价换来的是可追溯、可恢复、可解释的承诺边界。两级 readiness 允许 unknown 资源先形成安全草案，而不靠造数解锁；`quick` 可以降低 agent 深度，不能降低硬门。

## 它仍然不替代什么

v3 不替代投标人的资质核验、人员确认、报价审批、法律判断和最终签字，也不覆盖法定模板。它提供的是一套更可靠的技术标决策与生成底座，不是自动中标承诺。
